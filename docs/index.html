Vibe-Core v1.2.0

An ultra-low latency, file-based audio engine for Linux. Designed for FOSSHack 2026 with a focus on minimal resource overhead and Unix-style IPC.

The UAPI Architecture

Vibe-Core operates as a headless daemon. Interaction occurs through Memory-Mapped Control Nodes in /dev/shm/vibe. This allows any language (for example: Bash, Python, C) to control the engine by simply performing standard File I/O. The wrapper or user can find symlinks to /dev/shm/vibe/(file) in the local directory.

Control Endpoints (Write-Only)

1. Command Pipe (ctl)

A FIFO Named Pipe for real-time engine instructions.

echo "next" > ctl    # Skip current track
echo "prev" > ctl    # Previous track (Index - 2 logic)
echo "pause" > ctl   # Suspend audio device
echo "resume" > ctl  # Resume audio device

2. Seek Control (seek)

A FIFO Named Pipe for high-precision PCM stream positioning. The engine validates the timestamp against the stream length to prevent EOF crashes.

echo "30" > seek      # Jump to 30s
echo "120.5" > seek   # Jump to 2m 0.5s
echo "0" > seek       # Restart current song

3. Priority Injection (play_now)

Interrupt Handler: Passing an absolute path here immediately halts the current playlist, plays the target file, and then restores the previous playlist state.

echo "/home/user/music/urgent_vibe.mp3" > play_now

Stateful Nodes (Read/Write)

4. Volume Node (vol)

Text-based gain control and feedback. You can write to it to adjust gain or read from it to sync external UI sliders.

  • Write: echo "75" > vol
  • Read: cat vol (Returns current level)
  • Range: 0 - 100 (Unity) | 101 - 200 (Boost Mode)
cat vol               # Returns "50"
echo "125" > vol      # Applies 25% Digital Boost

Telemetry Nodes (Read-Only)

Node Description Type Data Format
state Engine playback status File playing | paused
now_playing Metadata of active stream File String (Filename)
head High-precision playback timer File Float (Seconds)

Performance Statistics

Testing performed on AMD E-350 (1.6GHz) running Arch Linux:

  • CPU Usage: ~2.1% (Stable)
  • RAM Footprint: Minimal
  • I/O Latency: Near-zero (RAM-backed via /dev/shm)
